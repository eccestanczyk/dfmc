ANALYTICS & TELEMETRY

Hub › Dev Tools › Analytics & Telemetry

Canonical plan (v1, July 2026) for hypothesis-driven telemetry: what we measure, where each number comes from, and which authored lever it moves. Purpose per design notes: rebalance drop rates, abilities, stats, general strength, and upgrade chances from observed progression speed — the default analytical lens is the non-buyer (self-found) player; every tracked metric must map to a named lever (a CSV column, a formula constant, or a Remote Config value); and balance truth comes from the server plane, never from client claims. Companion to Backend, Save & Anti-Cheat, which already mandates: UGS Analytics stays OFF (MAU-metered) — use free telemetry.

Principles

PrincipleRule
Server plane is truthBalance reads come from the Cloud Code boundary — server-computed rewards, server-rolled odds, replay-verified battles. Client-side events (sessions, UI, screens) are directional context, never balance evidence. Same doctrine as anti-cheat: the client reports events; the server derives numbers.
Free telemetry onlyPer the backend cost model, UGS Analytics is off (MAU-metered, ~$9–10k/mo at scale). Client plane: GameAnalytics free tier. Server plane: piggyback the existing Cloud Code flush — no new invocations, one outbound append per flush to owned storage.
Time is the anchorThe GDD already denominates the whole economy in battles (~30/h endgame). Every pacing question is answered in battles/hours, matching the design sheets, so observed reality and authored intent share one axis.
Hypothesis → metric → trigger → leverNo vanity metrics. Each register row names the number, the plane it comes from, the threshold that fires, and the authored lever it moves (creatures.csv grades, moves.csv powers, encounter_weights.csv, drop rates, odds ladders, Remote Config constants).
Segment before judgingNothing is interpreted without the buyer/non-buyer split and the realm split. A number that looks fine blended can hide a starving self-found cohort behind a market-fed one.
Numbers say what, feedback says whyTelemetry cannot distinguish "weak" from "feels bad." Every balance read pairs with the site feedback channel before a lever moves.
Privacy by defaultAnalytics ID = salted hash of SteamID64. The Settings menu already ships an Analytics opt-out toggle (on, opt-out) — both planes honor it. Export My Data / Delete Account flows include the analytics store and the feedback DB.

Two collection planes + the feedback loop

Three sources, one warehouse. The client plane exists from the prototype onward and costs nothing; the server plane arrives with roadmap step #16 and is the balance authority; the site plane (Steam-login feedback) is independent and can ship any time.

Fig. 1 — Collection topology. The server plane adds zero Cloud Code invocations: the analytics record is appended inside the flush that already runs every ~20 minutes. The client plane works today, in the prototype, before any backend exists.

The streak amplifier

Clearing a floor requires ceil(floor^0.65) consecutive wins, and a loss resets the streak. Expected battles to complete an n-streak at per-battle win rate p is E = (1−pⁿ) / ((1−p)·pⁿ) — exponential in n. This is why the win-rate curve per floor is the single most load-bearing chart in the plan: a 10-point win-rate dip that is invisible on floor 9 is a 6× cost multiplier on floor 99.

Win rate pFloor 9 (streak 5)Floor 25 (streak 9)Floor 55 (streak 14)Floor 99 (streak 20)

Expected battles to open the portal, by on-floor win rate. At ~30 battles/h endgame, 90% ≈ 2.4 h per deep clear; 80% ≈ 14 h. Boss floors bypass this (always 1 battle, key-gated).

Creature Feedback System (codex site)

Per design note: players sign into the codex site with their Steam account and leave direct, per-creature feedback. GitHub Pages is static, so auth and storage live in a small serverless layer; the site stays exactly where it is.

ElementSpec
AuthSteam OpenID 2.0. The Worker verifies the assertion with Valve, sets a signed HttpOnly session cookie, and stores only hash(SteamID64 + salt) — the same analytics_id, making telemetry joins native. No passwords, no accounts of our own.
WidgetOn creature.html: 1–5 rating + fixed tag chips (Too weak · Too strong · Boring kit · Fun to use · Confusing · Love the art · Art needs work) + optional comment ≤500 chars. One submission per account per creature, editable; edit history retained for trend.
Storage / APID1 table feedback(creature_id, sid_hash, rating, tags, comment, ts, version). GET /feedback/:creature returns public aggregates (avg, counts, tag histogram); POST requires the session cookie. Per-account rate limit, banned-word filter, admin delete.
WeightingNightly export joins feedback to telemetry: votes from players with ≥50 battles using the creature are weighted up; never discarded, just ranked. Feeds C6 quadrants in the weekly report.
DisplayAdmin mode: per-creature rating, tag histogram, and comment stream as an overlay in the codex (Dev Tools). Public aggregate display optional, later, behind approval.

Cadence & patch protocol

RhythmContents
Weekly balance reportScripted, non-buyer lens by default: floor win-rate curve + wall detector (D1/D2), creature quadrants (C1–C3), currency flows & blocked attempts (E4), gate tails (E3), boss/key friction (D4), container pressure (S4), early funnel + retention (S3/S2), feedback quadrant movers (C6).
Patch protocolEvery lever change is logged (date, lever, old→new, hypothesis ID it serves) and annotated on every chart. One lever per system per patch where possible; 7-day minimum before/after windows; the "butterfly problem" is real — expect side-effects, so re-read adjacent hypotheses after each change.
Integrity alarmsContinuous, not weekly: faucet deviation (E1), odds deviation (E3), faucet-rate/listing outliers (E7 — the backend's farm queue), error/crash rates (GA automatic).
QuarterlyRe-audit the proposed target bands against reality; retire hypotheses that never fire; add new ones from feedback themes.

Open flags & contradictions

#Flag
1Determinism contradiction. The GDD battle section and Backend v2 build on deterministic seeded replay (state + seed + move log); system-adventure-log.html states battles are not deterministic and rules out seed-replay. This plan works either way (battle aggregates are client-claimed + verified-subset-calibrated), but one ruling is needed — it decides whether per-battle damage could instead be replay-derived server-side at 100% trust.
2Boss turn targets disagree. HP budget section: kill in 8–13 turns; turn-target table: 10–15. C4/D4 alarm bands need one canonical window.
3Container capacities are undocumented. The GDD never states inventory slot caps (or whether eggs share the item inventory). Instrumentation is generic used/max per container the build exposes (inventory, hatchery, pens, crypt bag); capacities are read from the build, and S4 will surface whatever they are.
4All target bands above are proposals — win-rate floors, boss attempt p50, 70% velocity ratio, 14-day churn window, alarm widths. Nothing alarms until approved.
5Key drop rate and chests-per-floor are open GDD decisions. D4 and E5 are built to price them: instrument first, then author the numbers from observed friction.
6Site feedback adds a small serverless surface (Worker + D1 + Steam OpenID) — the first non-UGS backend piece. Deliberately isolated from game state; it can never grant anything.
