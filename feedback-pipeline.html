COMMUNITY FEEDBACK PIPELINE

Hub › Dev Tools › Community Feedback Pipeline

Strategy spec (v1, July 2026) for collecting, crunching, and acting on player feedback across creature designs, balance, moves, items, floors, bugs, and systems — sized for one developer. Extends the Analytics & Telemetry plan (Creature Feedback System, hypothesis C6). One sentence: every channel funnels into one warehouse, an AI pass agglomerates tickets into counted themes nightly, and the dev triages ~10 themes a week instead of reading 300 messages.

Principles

PrincipleRule
One warehouseEvery source — codex widgets, Discord forums, slash command, in-game button, Steam reviews — normalizes into one D1 tickets table. Nothing lives only in a Discord scroll. What isn't in the warehouse doesn't exist.
Meet players where they areFeedback SaaS vendors converge on one finding: standalone portals go unused; players vent where they already hang out. Our two "where they are" surfaces both exist — the codex (already the reference destination) and the Discord. No third destination is built.
Themes are the unit of workTickets are cheap and unbounded; the dev's attention is not. Triage happens at the theme level — a cluster of tickets about the same thing with a count attached. Work/discard decisions are made once per theme, never once per message.
Attention = unique voicesThe attention score counts distinct authors (tier-weighted) plus reactions — never raw message volume. One person posting ten times is one voice. Nothing is discarded for scoring low; it just ranks low. Same doctrine as the analytics feedback weighting.
Bugs and opinions are two tracksThe Mojang model: a structured bug tracker and an upvote-style feedback surface are separate intakes with separate templates. A bug needs version + floor + repro steps; an opinion needs a subject + text. Same warehouse, different forms, different console lenses.
Close the loop or it diesPeople file good reports only while reports visibly land. Status changes propagate back out: staff tags on Discord threads, a weekly digest post, ship-notifications in the original threads. Every source consulted treats this as the single highest-leverage habit.
Dev-only scoresCreature star aggregates are never public. This also removes the entire payoff of brigading — review-bombing a hidden number influences one reader who can see the raw data behind it.
Free tier onlyCloudflare Worker + D1 (already specced for analytics), GitHub Actions cron, one nightly small-model LLM pass (cents/month). Zero fixed cost, consistent with the backend cost doctrine.
Human decides, AI crunchesThe AI classifies, clusters, summarizes, and ranks. It never auto-files work, never auto-replies to players, never auto-discards. Community devs are unanimous that auto-ticketing every Discord message produces noise and bad-idea tickets; the human gate is the quality filter.

Intake surfaces

A — Codex site (extends the analytics Creature Feedback System)

ElementSpec
Creature starsOn creature.html: ★1–5 + fixed tag chips (Too weak · Too strong · Boring kit · Fun to use · Confusing · Love the art · Art needs work) + optional comment ≤500 chars. One rating per identity per creature, editable, history kept. Aggregates render only in admin mode — visitors see their own rating, never the average.
General feedback formFloating quill button on codex pages → kind selector (Creature design · Balance · Move · Item · Floor · System · Art · Other) + subject autocomplete fed from the codex CSVs (creatures, moves, items, floors) + text ≤1000 chars. Subject binding at intake is what makes per-subject agglomeration cheap and exact.
Bug report formSeparate form, separate template: build version · realm · floor · steps to reproduce · expected · actual, ≤1500 chars. Steam sign-in required (accountability + dedupe by author).
Later widgetsThe same star/tag widget generalizes by subject_ref to item.html, floor.html, and move rows once the creature version is proven. Not v1.

B — Discord restructure

ElementSpec
#suggestionsForum channel (not text). Required tags: Creature · Move · Balance · Item · Floor · System · Art · UI · Other. Default reaction 👍 + "Top" sort = native upvote board. Post guidelines (shown on every New Post): search first; one idea per post; title = subject + claim ("Omnichimera S3 — Eldritch kit feels passive").
#bug-reportsForum channel. Required tags: Crash · Battle · UI · Save · Economy · Visual · Other. Post guidelines embed the bug template (version / realm / floor / repro / expected / actual). Attachments enabled — screenshots and videos are half of every bug report's value.
Staff lifecycle tagsDev-applied only: Under ReviewPlannedIn ProgressShippedDeclinedDuplicateNeeds Info. These mirror theme status and are the visible half of the loop.
/feedback commandSlash command served by the Worker via Discord HTTP interactions — zero hosted bot. Modal with the same kind/subject/text fields, writes straight to D1. The low-friction path for players who will never write a forum post.
IngestionScheduled Worker (cron) polls both forums via Discord REST: new posts → tickets (title + body + tags + 👍 count + thread URL); reaction counts refreshed on poll. Replies are not auto-ticketed — the opening post is the ticket; discussion stays human-read via the thread link.

C — Deferred intakes

In-game report button (F4): one keypress in the Unity client posts to the same Worker endpoint with build, realm, floor, and recent battle context auto-attached — the friction-killer every playtest guide recommends. Rides the backend roadmap step, not before. Steam reviews (post-launch): nightly pull of new reviews via the public storefront API into tickets at low weight — review text is sentiment-rich but subject-poor, so it feeds themes rather than creating them.

Data model (D1)

TableColumns
ticketsid, source(site|discord|steam|ingame), kind, subject_ref, title, body, author_hash, tier, reactions, source_url, created, cluster_id, state(new|clustered|spam)
themesid, label, summary, kind, subject_ref, ticket_count, unique_authors, attention, trend_7d, status(inbox|considering|planned|in_progress|shipped|declined), decision_note, first_seen, last_seen
ratingssubject_ref, author_hash, tier, stars, ts — separate from tickets; powers the dev-only aggregates.

subject_ref is the agglomeration key and reuses codex identity: creature:DFMC-285, move:<id>, item:<id>, floor:41, system:breeding, or none. author_hash is the same salted-hash identity as analytics — the C6 feedback × telemetry join costs nothing.

The nightly AI pass

PassWhat it does
1 — ClassifyFor each state=new ticket: assign kind; extract subject_ref against the codex CSVs ("Omnichimera hits like a wet noodle" → creature:DFMC-285, Balance); translate non-English to English (community includes JP — no Mojang-style English-only wall); flag spam/empty → state=spam.
2 — ClusterEmbed title+body. Cosine above threshold joins the nearest open theme with matching kind/subject; otherwise a new theme is opened. The model writes or refreshes each touched theme's label and two-sentence summary with one representative quote. Themes never merge automatically — merge is a console button.
3 — Scoreattention = Σ unique-author tier weights + 0.1 × reactions; trend_7d = attention gained this week (what's heating up). Nothing is deleted for scoring low — low just ranks low.
OutputMonday digest: top 10 themes by attention, top movers by trend, new themes, totals — one crunched read. A dev-only copy lands in the console; a public copy (minus scores) is postable to #dev-updates.

Runs as a GitHub Actions cron calling the OpenAI API, writing back through an admin-token Worker endpoint. At a few hundred tickets a week on a small model this costs cents per month. Raw feedback text never enters the public repo — it stays in D1.

Triage console — feedback-inbox.html (build item)

Admin-gated page in Dev Tools. Themes ranked by attention; filters by kind, status, subject; each theme expands to its member tickets with source links (jump straight to the Discord thread). One-click status buttons across InboxConsideringPlannedIn ProgressShippedDeclined — Declined requires a one-line decision_note, which becomes the reply material when the loop closes. A ratings board lists lowest-rated / most-rated creatures from the dev-only star data, and the codex admin overlay shows per-creature aggregates in place. Once the telemetry plane lands, the C6 quadrant view (loved-but-weak / hated-but-strong …) joins here.

Closing the loop

MechanismSpec
Status syncConsole status change → Worker edits the staff tag on every linked Discord thread. Players watch their post move from Under Review to Planned without asking.
Ship-notifyWhen a theme goes Shipped, the bot replies once in each member thread: "Landed in patch X.Y — thanks." The single cheapest retention mechanic any of the studied tools offer.
Weekly digest postPublic digest copy to #dev-updates: what's top of mind, what moved status this week. Declined themes are listed with their one-line reason — visible "no" beats silent void.
Patch-note linkagePatch notes cite theme IDs; the analytics patch protocol already logs every lever change — the theme ID rides in that log entry, so charts, patches, and community asks stay cross-referenced.

Auth ruling — stars vs text

TierWhoCan doWeight
0 — AnonymousAny visitor; signed cookie UUID + IP rate capStar ratings only. No text of any kind.×0.25
1 — Steam signed-inSteam OpenID (one click, no password, no account of ours)Stars, tag chips, comments, feedback tickets, bug reports×1
2 — Verified climberTier 1 + telemetry join shows real playtime (e.g. ≥50 battles; ≥50 with the creature for creature feedback)Same as Tier 1, badge in console×2

Answer to the note: do not require Steam auth for stars; require it for anything with text. The reasoning is specific to this design: because aggregates are dev-only, brigading a hidden number has no audience and no payoff, so the classic reason to gate ratings disappears — while the friction cost of forced login on a one-click star is total. Text is different: comments and tickets need accountability, per-author rate caps, and author-dedupe, and OpenID costs the player one click. Anonymous stars are down-weighted, not trusted equally, and the tier is visible per-vote in the console.

Amendment flag: the Analytics & Telemetry page currently specs the creature widget as Steam-auth-gated for all input. This ruling amends it to the tiered model above (anonymous stars at ×0.25). Pending approval; on approval, the analytics page's Auth/Widget rows get a one-line edit.

Moderation, abuse & privacy

ConcernControl
Spam / floodingPer-identity rate caps (e.g. 20 tickets/day, 1 rating per creature, edits allowed); banned-word filter at the Worker; AI pass flags junk to state=spam. Admin delete is a shadow-discard — excluded from clustering, never surfaced back.
Sockpuppet starsCookie + IP caps blunt volume; ×0.25 weight bounds impact; the console can filter aggregates to Tier ≥1 in one click whenever a number looks manipulated.
Discord-side noiseServer verification level gates very new accounts; required tags stop untagged sludge; moderator "Duplicate" tag folds copies onto the original, matching the Mojang consolidation habit.
PrivacyOnly salted hashes of SteamID64 / Discord author IDs are stored — never raw IDs, matching the analytics identity doctrine. Free text is the one PII surface; the Export My Data / Delete Account flows already promised for analytics extend to tickets and ratings.

Cost

Cloudflare Worker + D1 free tier (the same instance the analytics plan provisions — 100k writes/day of headroom); GitHub Actions cron, free; nightly small-model classification at a few hundred tickets a week, cents per month on the existing OpenAI account; Discord slash command as HTTP interactions on the Worker — no hosted bot, no VPS.
