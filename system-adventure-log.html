ADVENTURE LOG

Other Systems › Adventure Log

A permanent, chapter-bound chronicle of everything that has ever happened on the account — hatches, evolutions, breedings, floor clears, boss kills, deaths, league events, and every battle fought. The log itself never forgets; only the depth of battle detail is pruned over time.

Architecture — events, not text

  • The log is an append-only event stream, not stored prose. Each entry is a compact record: {timestamp, event_type, param_ids[]} — e.g. {t, HATCH, DFMC-042}, {t, EVOLVE, DFMC-044 → DFMC-045}, {t, FLOOR_CLEAR, 17}.
  • Display text is rendered at view time from string templates plus codex CSV data. Renaming a creature or rewording a template retroactively fixes the entire history.
  • Cost per event: ~50–100 bytes. A heavy player producing 500 events/day for a full year is ~180k events ≈ 15–20 MB in SQLite — negligible.
  • Storage: one local SQLite database, append-only, indexed on date. No server round-trips to read the log.

Chapters

  • Chapters are dated views, not separate documents. A chapter is simply a date-range query (WHERE ts BETWEEN x AND y) against the single event table.
  • The chapter list UI shows one entry per active day (or per zone milestone on the same screen). Opening a chapter loads only that chapter's events, paginated — the full history is never held in memory.
  • Because chapters are queries, they cost nothing extra to store and can be re-cut later (per-day, per-zone, per-season) without touching the data.

Param IDs reference codex CSVs; the log stores no names or descriptions. Ordinary per-battle XP ticks are not logged as separate events — they live inside the battle summary.

Battle Logs (sub-item of the Adventure Log)

  • Battles are not deterministic, so full turn-by-turn data must be recorded — there is no seed-replay shortcut.
  • The adventure log itself holds only a battle summary row: participants, result, turn count, rewards, and a battle_id.
  • Full detail lives in a separate table keyed by battle_id: an array of compact turn-action structs {actor_id, move_id, target_id, damage, flags} at ~20–40 bytes each. A 30-turn battle ≈ 1–3 KB raw.
  • Each battle's turn array is serialized and gzip-compressed as one blob (3–5× reduction) before storage. Nothing is decompressed until requested.

Viewing a battle log

1 — Tap

The player taps the Battle Log button on a battle summary row in a chapter.

2 — Fetch & decompress

The blob is fetched by battle_id and decompressed in memory (<1 ms for a 1–3 KB blob) — only that one battle, never the rest of the history.

3 — Render

Turn structs are deserialized and rendered into readable turn-by-turn text via templates + codex data. Compression is invisible at the UI layer.

Retention policy

Battle classFull turn detailSummary row
Boss battlesKept foreverForever
PvP battlesKept foreverForever
Ordinary wild battlesLast N days / last N battles, then compacted awayForever
  • Compaction deletes only the detail blob; the summary row (and therefore the chapter narrative) is never lost — the log loses replay depth, not history.
  • After compaction, the Battle Log button on an old wild battle shows the summary card with a "detail expired" state instead of turn-by-turn text.

Storage budget

Scenario (20 s battles, nonstop)GrowthTime to 100 MB
1 KB/battle, uncompressed180 KB/h~570 h
3 KB/battle, uncompressed540 KB/h~190 h
With gzip (3–5×)—~600–2,800 h
With retention active, storage plateaus instead of growing linearly: the real cap is (battles retained in full detail) × ~0.5–1 KB compressed. Keeping the last 5,000 detailed wild battles plus all boss/PvP detail holds the entire system to single-digit MB permanently. The event stream itself grows unbounded but at 50–100 bytes/event it is never the constraint.
