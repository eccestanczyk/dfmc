BACKEND, SAVE & ANTI-CHEAT

Hub › Dev Tools › Backend, Save & Anti-Cheat

Canonical spec (v2, July 2026) for persistence, the server-authoritative economy, Steam integration, anti-cheat, and live-service cost. Supersedes the v1 draft. Stack per GDD: UGS (Auth, Cloud Save, Cloud Code, Leaderboards, Remote Config) + Steamworks.NET + Steam Inventory Service. One sentence: the client sends events and requests, never state; the server owns state and computes every result; Steam Cloud carries preferences only.

The principle & trust boundary

Local editing is not prevented — it is made inert. "I have 999,999 Scrap" is state and is never an input to any write. "Craft recipe R" / "I completed battle #8837" is a request or event, validated against the server's own copy of the data. The client keeps a disposable prediction cache for responsiveness; every Cloud Code return overwrites it with the authoritative snapshot. There is nothing to clean up after a cheat attempt because the illegal state never existed server-side.

Fig. 1 — Trust boundary and storage topology. Everything that gates loot, rank, trade, or death lives right of the boundary.

Data classification

Every persisted category maps to one authority, one store, one access class, one flush policy. Rule of thumb: if faking it affects loot, rank, trade, or death → Protected/Private on UGS, server-written. If faking it only affects the player's own convenience → client-local.

Access classes are the structural wall — a Protected/Private key physically cannot be written by the client SDK. Validation logic is the second layer. Keys are namespaced per realm (eternal.* / hardcore.* / seasonal.* / hcSeasonal.*); Codex and playerLevel are account-wide. State is written as consolidated documents per realm, not per-field keys — every extra key per flush is another billed write.

The economy transaction contract

Every spend — craft, refine, grade-up, soul grade, breed, enhancement — is the same atomic transaction computed against the server balance. Ordering is the entire guarantee: check, then atomically debit + grant, else reject whole. RNG rolls (refinement 100→1%, grading 100→0.5% ladders) resolve server-side, so a failed roll cannot be re-rolled by save-scumming — the currency is already gone on the server. Costs come from the same authored tables as the GDD (25×tier Scrap crafts, 3×lvl×tier refines, the 1/1/2/3/5/8/12/18/40 Ichor ladder, the five-material soul bundle).

Fig. 2 — Transaction gate. The "player edits Scrap and crafts ten items" scenario dies at the first read: server sees 40, all ten requests reject, zero items ever exist.

Rewards, batching & flush tiers

Rewards are derived server-side from completed battles; the client cannot assert an amount. Per event, Cloud Code recomputes Food (round(2×(1+lvl/20)×RarityMult)), XP with the level-penalty curve, and rolls the 10%/10% gear/enhancement drops from server tables — then applies one aggregated write. Batching groups events at the Cloud Code boundary: it changes how often we write, never who decides what is written.

Fig. 3 — Reward path. Immediate-tier events can never sit in an unflushed buffer; batched-tier loss on crash is bounded to a few minutes of personal grind — an availability tradeoff, never an economy hole. Replay reuses the GDD's determinism primitive (initial state + seed + committed move log).

Rate cap — variable battle speed

Battle duration is not constant: a floor-1 one-shot is far faster than a 5-turn endgame fight (the economy sheet's 280 Food/h at floor 1 already implies ~85+ battles/h there vs ~30/h endgame). So the cap is a curve derived from each battle's committed turn count, not a flat battles-per-hour number. Elapsed time is measured between Cloud Code's own flush timestamps — client clocks are never trusted.

Fig. 4 — Turn-count time floors. Guarantee: no account accrues faucet output faster than the design pacing curve for the content it fights. A headless client can forge 1,000 logs in 10 s of CPU — they credit at design pace, so the farmer gains exactly nothing over legal autoplay. Forged 1-turn logs against enemies the server-known party can't one-shot fail the sampled replay.

Log integrity — hash chain

Each battle event carries the previous event's hash. The server stores only the chain head per session; full logs stay client-side and are requested on audit. Streak and floor-record history becomes tamper-evident at ~zero storage cost.

Fig. 5 — Hash-chained event log. Editing, inserting, or deleting any past battle breaks every subsequent hash.

Concurrency — the actual historical dupe vector

Server-authoritative games get robbed through their transaction layer, not the client. The classic dupe: two simultaneous spend requests both read balance 100, both pass validation, both commit. Two mechanisms close it, both mandatory before trading exists: every request carries an idempotency key (retries return the cached original result), and every Cloud Save write uses the writeLock version token so concurrent writes to the same player document conflict and retry serially.

Fig. 6 — Optimistic concurrency + idempotency. 100 gold cannot become 200 gold of spending regardless of request timing.

PvP — commit-reveal, verify once

Rounds run peer-to-peer over Relay; the server is not in the per-round loop. Each side commits hash(move‖nonce), then reveals; the round's RNG seed derives from both nonces, so neither side can peek at the opponent's pick or steer a roll. Committed rounds persist before resolution renders (the GDD's anti-scum rule). At match end, one Cloud Code call replays the committed log, verifies it against both players' commitments, and writes the ladder — server-only. Timeouts and disconnects follow the PvP page: autoplay AI picks on expiry, five consecutive AI rounds forfeits.

Fig. 7 — Commit-reveal PvP with single end-of-match verification.

Hardcore death — one atomic commit

Death is confirmed server-side (the losing battle is replay-verifiable), then the whole pipeline applies as a single transaction. The death mark lives in a Private key the client cannot read, let alone clear; it is written atomically with the Bequest seal, so a wipe without a seal — or a seal without a mark — is structurally impossible. Sealing also transfer-locks the Steam-owned assets into the receiving crypt so nothing can be duped out mid-death. The HC-Seasonal → Hardcore double-migration (the GDD's highest-risk path) is a scheduled, idempotent server job at season end — never client-driven.

Fig. 8 — Hardcore death pipeline. Test all four realm paths and the season migration independently, per GDD critical risks.

Steam market & farm accounts

The mint is downstream of validation: a Cloud Code transaction confirms the asset was legitimately earned or paid for, then calls Steam GenerateItem with the publisher key and binds the instance ID back to Cloud Save. A faked balance never reaches the mint — the only thing touching the real-money market is the one thing that cannot be spoofed. Every trade path goes through a server-side transfer lock with a single-ownership check (in place before the marketplace opens, per GDD critical risks), and every item class carries a config-level tradeable flag so a compromised category can be frozen without a patch.

With client edits neutralized, the residual economic threat is farm accounts — F2P + legal 24/7 autoplay + a real-money market invites item-printing at account scale. Every farm account plays "legitimately," so this is fought with gates and telemetry, not client tech:

What other games teach

CoC's client is so purely a view that full private-server reimplementations exist; modding the client against the real server yields nothing.Exactly this page's model. Events in, computed state out.
Diablo 3 gold dupe (2013)A patch bug in the auction house — not a client hack — let players duplicate gold into the trillions. Blizzard took the AH offline, audited every transaction since the patch, banned the 415 accounts that exploited it, recovered ~85% of the gold, and chose targeted removal over a rollback.The transaction layer is the attack surface. Keep a per-account transaction audit trail from day one, keep a market kill-switch, and design so surgical removal is possible — a live-economy rollback is not a real option.

Keep all liquidity inside the Steam Market (Valve's fraud detection, trade holds, and authenticator friction protect us for free) and keep the per-class tradeable flag revocable by config — Valve's exact lever.

Our autoplay design removes the botting arms race entirely: grinding automation is a feature, so "bot detection" reduces to the rate cap + farm-account gates above.

Live-service cost model

UGS pay-as-you-go unit prices (July 2026): Cloud Code $1.50/M invocations + $0.00002 per compute-second (1M invocations + 20 compute-hours free/mo); Cloud Save $2.50/M writes, $0.90/M reads, $0.50/GiB stored (1M + 1M + 5 GiB free); egress $0.25/GiB; Relay $0.16 per average monthly CCU after the first 50; Leaderboards currently "free for a limited time".

Load model: autoplay per the GDD; flush every 20 min + ~1 immediate event/h ⇒ 4 Cloud Code invocations/h/CCU, each 1 consolidated-doc read + 1 write, ~50 ms compute, ~1 KB delta response; CCU treated as sustained 24/7 (autoplay worst case, 730 h/mo); PvP = 2% of CCU, commit-reveal, 2 verify calls/match, Relay transport; storage 50 KB per stored account.

≈ $0.02–0.03 per CCU per month. Cheap because the client simulates and the server only validates — this is a key-value store plus arithmetic, not game servers. Contrast the naive build (per-battle invocations, per-field writes, currencies on the UGS Economy service): it computes to $450k+/mo at 500k CCU, dominated by Economy's $35/M writes. Implementation order

This spec is roadmap step #16 — Backend authority layer (Simplified Static). Groundwork lands earlier: the save-skeleton key schema with #1 Data architecture, and seed + move-log persistence with the #6/#10 battle system (replay verification needs them). Step #16 delivers Auth, access classes, the transaction contract, reward validation, the rate cap, and idempotency + writeLock; then #17 Realms & death consumes the atomic death pipeline, #18 PvP the commit-reveal verifier, and #19 Steam integration the mint + transfer lock. Open decisions: publisher-key host (Cloud Code direct vs thin Lambda), Soul Gem tradeability, death-marked creature tradeability, and whether settings mirror to UGS for the future mobile port (Steam Cloud is PC-only).
