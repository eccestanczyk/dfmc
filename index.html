Concept

A creature-collecting roguelite RPG where players climb an ever-ascending tower floor by floor. Each floor is a simple map where the Collector triggers battles until a win threshold opens the portal to the next floor, with manual chests scattered throughout. All metagame systems (breeding, crafting, farming, hatchery, crypt) are accessible as UI panels at any time during PvE — no physical base.

Game modes

Eternal
Soft death. Character stays in league, no reset. Houses the Eternal crypt (UI panel). Receives crypt contents from seasonal migrations.
Hardcore
On death, character is wiped to level 1 and stays in Hardcore. All items, creatures, and class XP (as soul gems) go to the Eternal crypt.
Seasonal
Soft death. Timed league. On season end, character migrates to Eternal. Seasonal crypt contents delivered to Eternal crypt.
HC Seasonal
On death, character wiped to level 1 and stays in HC Seasonal. Items go to Seasonal crypt. On season end, HC Seasonal crypt delivered to Hardcore crypt.

Ladders

  • PvP ladder — ranked 3v3 combat, tracked per season and all-time
  • Progress (floor) ladder — highest floor reached, tracked per season and all-time

Art pipeline

Dark fantasy realistic painting style generated with AI tools (ChatGPT/DALL-E or Midjourney). A locked style-guide prompt must be established before mass generation begins. Flow: Generate → Upscale → Background removal → Import to Unity via Addressables.

Character system

One character per league, always

Every account has a character in every league they have unlocked, starting at level 1. The character slot is permanent — death does not remove or demote the character. On hardcore death, the character is wiped back to level 1 and remains in the same league. Only their items, creatures, and class XP are lost.

Character level vs class level

The character has a character level — a global level aggregated from XP across all classes. Character level scales base stats (HP, stamina, resistances). Each class has its own class level tracked separately, unlocking skill tree nodes and class-specific bonuses. Switching classes does not reset others.

Skill tree

Each class has a Path of Exile–style passive skill tree — a large interconnected graph of nodes. Players spend skill points (earned on class level-up) to unlock adjacent nodes only.

  • Node types: stat (flat bonus), modifier (% bonus), keystone (build-defining passive)
  • Must path from starting node outward — no jumping to isolated nodes
  • Respec costs resources but is always possible
  • Class specializations gate deeper sub-trees inaccessible without specialising

Soul gems

When a hardcore character is demoted, their soul fractures. Each class they trained echoes forward as a Soul Gem, deposited in the crypt for the surviving character to reclaim.

One Soul Gem is generated per class the character had any XP in. Rank scales with XP amount. Consuming a gem grants that XP to the character's matching class.

Core loop

Multi-character tab system

One game window contains up to 4 tabs — one per active league character. All active characters simulate simultaneously inside a single executable instance. There is no offline progression; being in-game is the only way to advance.

Manual mode
Default state. Player directly controls the character on the selected tab. Autoplay must be explicitly enabled to hand off control.
Autoplay mode
Must be enabled per character. Any tab can run on autoplay — including the currently selected one. Disable autoplay on the active tab to return to manual control.
  • Autoplay is off by default — players opt in per character
  • Background tabs with autoplay disabled are paused/idle
  • Background tabs with autoplay enabled continue simulating
  • Switching to a tab does not automatically disable autoplay — player must turn it off manually to take control
  • Tabs only exist for leagues the Collector has an active character in — new accounts start with 1 tab
  • Closing the game window pauses all characters
  • No offline rewards — open game only
  • All tabs share one executable process — not a performance concern given the 2D turn-based nature of the game

Unity implementation note: Multiple simultaneous scenes use additive scene loading — a well-documented Unity pattern. Straightforward to implement.

Overworld movement

Classic top-down grid movement. 4 directions, one tile per step — identical to Generation II Pokémon games (SoulSilver/HeartGold). Unity Tilemap system with orthographic camera.

Movement rules
  • Face direction on input even if blocked by collision
  • Raycast or collision check ahead before stepping
  • Animation states: idle and walk for all 4 directions
  • Random encounter triggers on step count + RNG roll

Map design

Maps are small and simple — depending on floor theme, the Collector triggers battles one of three ways. No elaborate handcrafted circuits or secret areas. The Collector engages with the floor's encounter method until the win condition is met.

🌿 Outdoors
Battles trigger only while standing on tall grass tiles. Autoplay finds a grass patch and walks around within it, avoiding walls.
🕳️ Cave
Battles can trigger anywhere on the floor. Autoplay walks around the full map continuously, avoiding walls.
🎣 Fishing
Battles trigger by fishing at a water tile, not by walking. Autoplay parks adjacent to water and repeatedly fishes in place.

Each floor's encounter type is fixed by its theme. The win condition (10 battles), boss key drop, and chest system apply identically across all three types — only the trigger method and autoplay behavior differ.

🏆 Win condition
Win 10 battles on a floor to open the portal to the next floor. Battles won is tracked per floor visit.
👑 Boss floors
Every 5th floor is a boss room. Entry requires consuming a Key, which has a chance to drop from any battle across the previous 4 floors.

Boss key

A single Key is required to enter a boss floor, accessible via the portal or directly through the map selector. Any battle won on the 4 floors preceding a boss has a chance to drop that floor's Key. The Key is consumed on boss entry.

Chests

Chests are scattered around each map as manual-only, one-time pickups — autoplay does not collect them. Each chest can only be opened once per floor visit cycle. The map selector displays chest collection progress per floor as x/x collected.

Floor structure

ElementDetails

Turn structure

Speed stat determines order within a turn. Status effects queued (poison, stun, burn, etc.) and resolve after move resolution.

Open decision: Simultaneous or sequential turns? Define before battle system implementation begins.

PvP sync

Both players submit actions simultaneously. Server resolves outcomes. Unity Relay for connection. Never resolved client-side.

Anti-cheat: Server-authoritative or server-verified only for PvP.

Crypt

The crypt is a UI panel accessible at any time during PvE — there is no physical base or chest to walk up to. Two crypts exist, one per hardcore league type:

Eternal crypt
Accessible via UI in the Eternal league. Receives items from Hardcore deaths. Also receives Seasonal crypt contents at season end.
Seasonal crypt
Accessible via UI in the Seasonal league. Receives items from HC Seasonal deaths. At season end, contents are transferred to the Hardcore crypt.

What goes into the crypt on death

  • All equipped gear at time of death
  • All inventory items
  • Soul gems (one per class with any XP, rank determined by XP amount)
  • All tamed creatures — creatures are treated as items for this purpose

Hardcore death pipeline

  1. 1Character dies in Hardcore league
  2. 2Character flagged: diedOnHardcore = true, deathFloor and timestamp recorded
  3. 3All class XP converted to soul gems — one per class with any XP, rank by XP amount
  4. 4All gear, inventory, soul gems, and creatures deposited into Eternal crypt chest
  5. 5Character reset to level 1, remains in Hardcore league with death mark on status sheet

HC Seasonal death pipeline

  1. 1Character dies in HC Seasonal league
  2. 2Character flagged: diedOnHardcore = true, deathFloor and timestamp recorded
  3. 3All class XP converted to soul gems — one per class with any XP, rank by XP amount
  4. 4All gear, inventory, soul gems, and creatures deposited into Seasonal crypt chest
  5. 5Character reset to level 1, remains in HC Seasonal league with death mark

Metagame systems

All metagame systems are accessible as UI panels at any time during PvE floors, including during battles. They are locked during PvP. There is no physical base — everything is menu-driven.

🧬 Breeding
Two parent creatures fed Gruel produce an egg via UI. Higher grade creatures require more Gruel. Higher grade parents increase odds of higher grade eggs. No grade cap on bred eggs.
🔧 Crafting
Recipe list in UI. Select recipe, confirm materials, receive output. Materials sourced from battle rewards and chests.
🌱 Farming
Timer-based UI system. Plant a crop, return after a set duration to harvest. Yield renders into Gruel or crafting materials.
🥚 Hatchery
Egg incubation managed via UI. Set eggs to incubate, return after the timer. Hatchery level (upgraded via crafting resources) improves grade odds and slot count.

Creature acquisition

There is no capture mechanic. All creatures come from eggs — either dropped in combat or produced via breeding. Every egg must be incubated at the base before hatching. Egg grade determines the stat range of the hatched creature and becomes the creature's grade.

🥚 Wild eggs
Drop from combat encounters. Grade range: Worn through Legendary only — no wild eggs above grade 6. Grade determines stat range at hatch.
🧬 Bred eggs
Produced via the Breeding UI panel by selecting two parent creatures and spending Gruel. No grade cap — can produce up to Void. Higher grade parents significantly increase odds of higher grade output. Hatchery level also increases the chance of higher grade eggs independently of parent grade.
⭐ Starter creatures
The player's very first creatures are selected from a set of 3 options before any combat. Grade and stats of starters are TBD.

Gruel

Gruel is the unified food resource used to breed creatures. Higher grade creatures require more Gruel per breeding attempt. Three sources all render into the same Gruel pool:

SourceHow it becomes Gruel
EggsUnwanted eggs can be rendered into Gruel at the base
CropsFarmed crops rendered into Gruel at the base
Creature meatDefeated creatures drop meat, rendered into Gruel at the base

Incubation & hatchery

All eggs must be incubated before hatching, managed via the Hatchery UI panel. The hatchery has two upgrade vectors, improved by spending crafting resources:

Hatchery upgradeEffect

Mutant variants

Any egg has a small chance to hatch as a Mutant — a creature with an alternate color scheme caused by a rare pigmentation mutation. Mutant status is purely cosmetic: it grants a medal on the creature sheet and changes the creature's color palette, but has no effect on stats or gameplay. Chance scales with egg grade. Completing the Codex adds a permanent +0.1% bonus to all mutant hatch chances.

Codex

The Codex is a Pokédex-style catalogue of every creature the Collector has encountered or acquired. It serves as both a reference tool and a progression record.

Entry states

StateConditionInformation visible
UndiscoveredNever encounteredSilhouette only, no data
SeenEncountered in battle at least onceName, floor levels found, basic stats ranges visible
AcquiredHatched from an egg of this creature typeFull entry — all moves, full stat ranges, evolution chain, egg grade ranges

Entry contents (full)

📍 Encounter data
Floor levels the creature is found on, encounter type (Outdoors / Cave / Fishing), and whether it appears as a wild encounter or boss.
⚔️ Moves
All possible moves the creature can have, with damage type, effect, and unlock level. Moves the player has seen used in battle are marked separately from full unlock.
📊 Stat ranges
Min and max possible values for each stat (HP, Attack, Defense, Speed, etc.) across all grades. Higher grade creatures have higher stat ceilings.
🧬 Evolution chain
Full evolution tree for the creature line, showing trigger conditions (level, item, bond). Evolution stages link to their own Codex entries.

Codex scope

  • One Codex per account, shared across all leagues — seeing a creature in any league updates the Codex globally
  • Acquired status is league-specific — owning a creature in Eternal does not mark it as acquired in Hardcore
  • The Codex is never wiped on death — it persists permanently and cannot be reset

Open decision: Should the Codex be accessible from the base only, or viewable from the floor UI at any time (including during autoplay)? Also: does the Codex track personal best stats of owned creatures, or only the theoretical range?

Creature souls

Each piece of gear can have the soul of a creature attached to it, granting special effects or attributes. Every creature has a different effect for each gear slot — the same creature attached to a helmet vs boots produces different bonuses.

Grade determines how many gem, rune, and enchantment slots a piece of gear has. Equipment is also tiered every 10 character levels. Refinement

Any piece of gear can be refined from +1 up to +10, increasing its power. Refinement uses Scrap, acquired by dismantling equipment. Refined equipment yields more Scrap when dismantled, creating a recycling loop.

RuleDetail
CurrencyScrap — gained by dismantling gear
Cost scalingHigher equipment tier and higher current refinement both increase Scrap cost
Success chanceDecreases at each level — +9 to +10 is extremely rare
On failureNo progress, no destruction — Scrap is consumed but gear is unaffected
Dismantling refined gearYields more Scrap than base gear — scales with refinement level

Grading up

Gear can be upgraded to the next grade (e.g. Rare → Epic) using Ichor. Functions identically to refinement in terms of cost scaling, failure chance, and no-destruction on failure.

RuleDetail
CurrencyIchor
Cost scalingHigher current grade increases Ichor cost
Success chanceDecreases at higher grades — upgrading into Primordial and Void is very rare
On failureNo progress, no destruction — Ichor is consumed but grade is unchanged

Open decision: How is Ichor acquired? Possible sources: drops from bosses, dismantling high-grade gear, crafting, or rare battle/chest drops.

Economy & marketplace

Monetization model

  • Free to play on Steam
  • Microtransactions via Steam / Unity Economy
  • Battle pass per season
  • Steam takes 5% market fee + configurable game fee (up to ~10%)

Cosmetics-only monetization is safer for player reception and avoids regulatory issues in certain markets. Decide scope early.

Steam marketplace integration

Tradeable items and creatures are Steam Inventory items. Ownership lives on Steam; gameplay data lives on the backend keyed to instance IDs.

Item grant flow (server-side only)

  1. 1Player completes breed/craft action
  2. 2Client sends request to cloud function (AWS Lambda / PlayFab CloudScript)
  3. 3Server validates action is legitimate
  4. 4Server calls Steam GenerateItem API with publisher key
  5. 5Steam issues item to player inventory
  6. 6Confirmation returned to client

Never grant Steam inventory items from the client. Server-side only with publisher key validation.

Player logging system

Two separate logs give players a transparent record of what happened, both implemented after core gameplay systems (battle, leveling, items) are in place — building the export pipeline against placeholder data would mean redoing it once real systems exist.

Log typeContents
Event logHigh-level player events: died, entered battle, completed battle (with XP and items acquired), acquired boss key, progressed to next floor
Battle logDetailed per-battle record of every move and result, scoped to a single battle
  • Both logs are downloadable as plain text files
  • PC: saved to a standard local file location, directly accessible to the player
  • Mobile: exposed via a native share/download UI since mobile platforms can't expose a raw file path the way PC can
  • Implementation deferred until battle system, leveling, and item systems exist — logging hooks should be added naturally as those systems are built, not retrofitted

Build roadmap

Recommended build order — each phase should be playable before moving to the next.

  1. 1Data architecture — all ScriptableObjects and save system skeleton
  2. 2Single-floor overworld — grid movement, collision, camera, tilemaps
  3. 3Encounter trigger — battle scene entry, minimal 1v1 battle, return to overworld
  4. 4Full battle system — expand to 3v3, moves, status effects, animations
  5. 5Floor progression — boss gate, stairs, floor transition, multiple floors
  6. 6Floor win condition & chests — battle win counter, portal unlock, manual one-time chests, chest tracking on map selector
  7. 7Boss key & boss floors — key drop chance on pre-boss floors, key consumption, boss floor access via portal or map selector
  8. 8Autoplay & multi-tab system — pathfinding, config options, death handler, simultaneous multi-character simulation
  9. 9Character & class system — class XP, character level aggregation, PoE skill trees
  10. 10Codex — creature catalogue, seen/acquired states, stat ranges, move lists, evolution chains
  11. 11Metagame UI — breeding, crafting, farming, hatchery — all as in-game UI panels accessible during PvE
  12. 12Player logging system — event log and battle log, hooked into existing battle/leveling/item systems, file export for PC and mobile share UI
  13. 13Leagues & death pipelines — all four death paths, both crypts, soul gem generation, season migration
  14. 14PvP & ladders — matchmaking, relay, leaderboards
  15. 15Steam integration — Steamworks.NET, inventory, marketplace
  16. 16Monetization — economy service, battle pass, item grants
  17. 17Steam release — store page, F2P config, achievements

Critical risks

PvP anti-cheat — server-authoritative resolution required. Design this before any PvP work begins.

Art consistency — lock style-guide prompt before mass generation.

Death pipeline integrity — four death paths + two crypt systems + season migration must all be tested exhaustively. HC Seasonal double-migration is highest risk.

Seasonal migration — crypt chest delivery mechanism must be rock solid before launch.

Trade duplication exploits — implement transfer lock before marketplace goes live.

Future features (cut from current scope)

Rogue Collectors — hostile NPC trainer encounters with patrol routes and line-of-sight triggers, originally part of floor structure. Removed from active scope to keep early development focused. May be reintroduced post-launch or in a content update.

Gathering nodes & secret areas — the original handcrafted circuit map design with resource nodes, HM-gated secret areas, and a node-farming layer has been replaced by the simpler battle-win-threshold map design. All equipment and crafting materials now come from battle rewards and chests only. May be reconsidered as a future map variant or content layer.

Physical base — the original design included a physical base the Collector could walk around in, with tile-based farming, craftings stations, a hatchery building, and a crypt chest to interact with. This has been replaced entirely by UI panels accessible during PvE floors. A physical base may be reintroduced in a future content update as an optional social/cosmetic space.

Open decisions

  • Cosmetics-only vs convenience items in shop
  • Breed count limit per creature
  • Which AI art tool to standardize on
  • Whether death-marked creatures are tradeable on the marketplace
  • Soul Gem XP thresholds for each rank (F through S)
  • Whether Soul Gems are tradeable on the Steam marketplace
  • Class specialization names and tree structures for all three classes
  • Number of chests per floor (the "x" in the x/x chest tracker)
  • Key drop rate per battle on pre-boss floors
  • Whether chest contents reset along with the 10-battle win counter on floor re-entry, or remain collected permanently

A creature's floor is its level. Which evolutionary stage can appear is gated globally by floor number, not by zone — zones are still fixed 10-floor thematic/environment blocks, but stage eligibility ignores zone boundaries entirely.

All 42 creature lines are full 3-stage — no exceptions. Rarity (C/UC/R/SR/SSR) is a fixed property of the line, independent of stage, since every creature on a given floor is necessarily the same stage. 42 lines split: 13 Common, 13 Uncommon, 8 Rare, 4 Super Rare, 4 Super Super Rare.

Encounter weights — SSR/SR odds scale up over the run

Two separate tables work together. codex/floor_creature_rarity.csv says which species are eligible in each tier on a given floor (always 3C/3UC/2R/1SR/1SSR — fixed). codex/encounter_weights.csv says the actual odds of an encounter rolling into each tier on that floor — and those odds move as the run progresses, so flagship lines are technically always "on the table" but genuinely hard to actually meet early on.

Superseded note below — kept for context, but rarity is now per-line (see level bands above), not per-stage.

Not the same scale as equipment/egg Grade above. Every floor has exactly 10 encounter slots: 3 Common, 3 Uncommon, 2 Rare, 1 Super Rare, 1 Super Super Rare. Data lives in codex/floor_creature_rarity.csv. A line's strongest evolution is typically SSR on a zone's earliest floors, shifting down to SR by that zone's later floors.

Creature grading design principle

Each creature line has one fixed set of 4 identity grades — Offense / Defense / Utility / Speed, scale 1–10 — that stays constant across every evolution stage in that line. Evolution scales actual battle stats later; it does not change the identity shape.

Image generation conventions

  • Default background: solid chroma green (0,255,0). Use magenta (255,0,255) instead for Fungal/Flora/Insect-typed creatures, since those legitimately render green.
  • 3-stage workflow: design stage 2 first as the core concept, then generate stage 1 and stage 3 as new images using stage 2 as a visual reference — never as an "edit" of stage 2.
  • No repeated core motifs between different creature lines. If a theme must repeat, it needs an added twist unique to that creature.
