Concept

A creature-collecting roguelite RPG where players climb an ever-ascending tower floor by floor. Each floor is a self-contained map with gathering nodes, random battles, NPC trainers, and a boss. The metagame revolves around breeding, crafting, farming, and base-building between runs.

Game modes

Eternal
Permanent progression. Dead hardcore characters arrive here with a death mark. Houses the crypt.
Hardcore
Death is permanent. Character flagged and migrated to Eternal. Gear, soul gems, and creatures deposited in the crypt.
Seasonal
Timed league with a separate ladder. At season end, all characters migrate to Eternal.
Hardcore Seasonal
Death sends character to Seasonal regular (not Eternal, no crypt). Both hardcore and seasonal rules apply.

Ladders

  • PvP ladder — ranked 3v3 combat, tracked per season and all-time
  • Progress (floor) ladder — highest floor reached, tracked per season and all-time

Art pipeline

Dark fantasy realistic painting style generated with AI tools (ChatGPT/DALL-E or Midjourney). A locked style-guide prompt must be established before mass generation begins. Flow: Generate → Upscale → Background removal → Import to Unity via Addressables.

Character system

Character level vs class level

Every account has one character with a character level — a global level aggregated from XP across all classes. Character level scales base stats (HP, stamina, resistances). Each class has its own class level tracked separately, unlocking skill tree nodes and class-specific bonuses. Switching classes does not reset others.

Skill tree

Each class has a Path of Exile–style passive skill tree — a large interconnected graph of nodes. Players spend skill points (earned on class level-up) to unlock adjacent nodes only.

  • Node types: stat (flat bonus), modifier (% bonus), keystone (build-defining passive)
  • Must path from starting node outward — no jumping to isolated nodes
  • Respec costs resources but is always possible
  • Class specializations gate deeper sub-trees inaccessible without specialising

Hardcore death — crypt system

When a hardcore character dies, their soul fractures. Each class they trained echoes forward as a Soul Gem, left in the crypt for future inheritors.

Crypt is Eternal-only. Only Hardcore (eternal) deaths trigger the crypt. Hardcore Seasonal deaths go to Seasonal regular — no crypt deposit, no soul gems.

On an Eternal-bound hardcore death, the following are sent to the crypt chest in the base's graveyard:

Item typeDetails
Soul GemsOne per class the character had any XP in. Rank scales with XP. Consuming grants that XP to a living character's matching class.
Equipped gearAll gear worn at time of death.
Inventory itemsEverything in the character's inventory.
Dead hardcore creaturesAll tamed creatures that died in hardcore on that account.

Item description shows exact XP granted on consume. Rank thresholds (F through S) are TBD — to be balanced during development.

Crypt chest rules

  • Account-wide and persistent — never resets
  • Accumulates items from all deceased eternal-bound hardcore characters
  • Creatures in the crypt retain stats, moves, and death mark — can be revived into Eternal
  • Soul gems are consumable — once used, gone
  • Gear is immediately usable by living characters

Core loop

Overworld movement

Classic top-down grid movement. 4 directions, one tile per step — identical to Generation II Pokémon games (SoulSilver/HeartGold). Unity Tilemap system with orthographic camera.

Movement rules
  • Face direction on input even if blocked by collision
  • Raycast or collision check ahead before stepping
  • Animation states: idle and walk for all 4 directions
  • Random encounter triggers on step count + RNG roll

Floor structure

Random battlesStep-counter trigger, encounter table varies by floor

Healing

Healing items can be used between fights on a floor (manually or on auto). Not usable mid-battle.

PvP format

  • 3v3: 1 avatar + 2 creatures per side
  • Best of 3 — no sideboarding between games
  • Team locked at match start
  • Results post to PvP ladder

Architecture

BattleManager (state machine)
├── PlayerSide: [Avatar] [Creature1] [Creature2]
├── EnemySide: [1–N enemies] (PvE) / [Avatar + 2] (PvP)
├── TurnManager
├── ActionQueue
├── AnimationController
└── BattleResultHandler

Turn structure

Speed stat determines order within a turn. Status effects queued (poison, stun, burn, etc.) and resolve after move resolution.

Open decision: Simultaneous or sequential turns? Define before battle system implementation begins.

PvP sync

Both players submit actions simultaneously. Server resolves outcomes. Unity Relay for connection. Never resolved client-side.

Anti-cheat: Server-authoritative or server-verified only for PvP.

Creatures & characters

  • Characters level up and spend skill points on a PoE-style tree per class
  • Creatures level up, spend skill points, and have evolution chains
  • Evolutions triggered by level, item, or bond — swap CreatureData ScriptableObject, preserve stat history

Leagues & death system

Death destination by league

LeagueOn deathCryptLadder
EternalSoft — respawn at baseNoAll-time floor
HardcorePermanent → migrates to Eternal + crypt depositYesAll-time floor
SeasonalSoft — respawn at baseNoSeason floor + PvP
HC SeasonalPermanent → migrates to Seasonal regularNoSeason floor + PvP

The crypt is triggered only by Hardcore (eternal) deaths. Hardcore Seasonal deaths migrate to the current Seasonal regular league — no crypt, no soul gems, gear stays with the character.

Hardcore (eternal) death pipeline

  1. 1Character dies in Hardcore league
  2. 2Character flagged: diedOnHardcore = true, deathFloor and timestamp recorded
  3. 3Soul Gems generated — one per class with any XP, rank determined by XP amount
  4. 4Equipped gear, inventory, and all account's dead hardcore creatures deposited in crypt chest
  5. 5Character migrated to Eternal league with permanent death mark on status sheet
  6. 6Crypt chest accessible to all living characters on the account

Hardcore Seasonal death pipeline

  1. 1Character dies in Hardcore Seasonal league
  2. 2Character flagged: diedOnHardcore = true, deathFloor and timestamp recorded
  3. 3Character migrated to current Seasonal regular league with death mark
  4. 4Gear and inventory stay with the character — no crypt deposit, no soul gems
  5. 5At season end, character migrates from Seasonal regular to Eternal as normal

Season lifecycle

  1. 1Season start date set in Unity RemoteConfig
  2. 2Players create fresh characters in Seasonal or HC Seasonal
  3. 3Ladder tracks floor progress and PvP rank
  4. 4Season ends → all surviving seasonal characters migrate to Eternal with season badge
  5. 5HC Seasonal dead characters (already in Seasonal regular) also migrate to Eternal
  6. 6Seasonal ladder archived, new season begins

Test all four death pipelines independently. The HC Seasonal → Seasonal regular → Eternal double-migration path is the most likely source of data bugs.

Metagame systems

These systems run between tower runs and constitute the long-term engagement loop.

🧬 Breeding
Two parent creatures produce offspring. Stat inheritance rules determine base stats. Egg timer before hatch. Breed count per creature may be limited.
🔧 Crafting
Recipe ScriptableObjects. Inventory check at craft time. Outputs items or equipment. Materials sourced from gathering nodes and farming.
🌱 Farming
Tile-based at player base. Growth timers per crop. Yield feeds crafting inputs. Farm plot count expands via base upgrades.
🏗️ Base upgrading
Building levels unlock features: more farm plots, more breeding slots, larger storage, crafting stations. Resource sink for gathered materials.

Skill trees

Both characters and creatures have PoE-style passive skill trees as graph ScriptableObjects. SP earned on level-up. Nodes unlock adjacent nodes only. Respec always possible at a resource cost.

Economy & marketplace

Monetization model

  • Free to play on Steam
  • Microtransactions via Steam / Unity Economy
  • Battle pass per season
  • Steam takes 5% market fee + configurable game fee (up to ~10%)

Cosmetics-only monetization is safer for player reception and avoids regulatory issues in certain markets. Decide scope early.

Steam marketplace integration

Tradeable items and creatures are Steam Inventory items. Ownership lives on Steam; gameplay data lives on the backend keyed to instance IDs.

Ownership model
SteamInventory
└── Owns CreatureToken #ItemDef_042, InstanceID 8837623

Your Backend
└── CreatureInstance_8837623 {
      level, moves, stats,
      diedOnHardcore, breedCount
    }

Item grant flow (server-side only)

  1. 1Player completes breed/craft action
  2. 2Client sends request to cloud function (AWS Lambda / PlayFab CloudScript)
  3. 3Server validates action is legitimate
  4. 4Server calls Steam GenerateItem API with publisher key
  5. 5Steam issues item to player inventory
  6. 6Confirmation returned to client

Never grant Steam inventory items from the client. Server-side only with publisher key validation.

Save system

  • Eternal / Hardcore / Seasonal / HC Seasonal are separate save profiles
  • Character level and all class levels saved per character
  • Crypt chest is account-wide, Eternal-only, never resets
  • HC Seasonal dead characters tracked in Seasonal regular until season end
  • Ladder and trade data: cloud save (anti-cheat)
  • Hardcore death flag persists permanently and cannot be removed

Build roadmap

Recommended build order — each phase should be playable before moving to the next.

  1. 1Data architecture — all ScriptableObjects and save system skeleton
  2. 2Single-floor overworld — grid movement, collision, camera, tilemaps
  3. 3Encounter trigger — battle scene entry, minimal 1v1 battle, return to overworld
  4. 4Full battle system — expand to 3v3, moves, status effects, animations
  5. 5Floor progression — boss gate, stairs, floor transition, multiple floors
  6. 6Gathering & trainers — node collection, NPC patrol, one-time battles
  7. 7Autoplay — pathfinding, config options, death handler
  8. 8Character & class system — class XP, character level aggregation, PoE skill trees
  9. 9Metagame — breeding, crafting, farming, base building
  10. 10Leagues & death pipelines — all four death paths, crypt chest, soul gem generation
  11. 11PvP & ladders — matchmaking, relay, leaderboards
  12. 12Steam integration — Steamworks.NET, inventory, marketplace
  13. 13Monetization — economy service, battle pass, item grants
  14. 14Steam release — store page, F2P config, achievements

Critical risks

PvP anti-cheat — server-authoritative resolution required. Design this before any PvP work begins.

Art consistency — lock style-guide prompt before mass generation.

Death pipeline integrity — four separate death paths must be tested exhaustively. HC Seasonal double-migration is highest risk.

Seasonal migration — plan the full data pipeline before launch.

Trade duplication exploits — implement transfer lock before marketplace goes live.

Open decisions

  • Simultaneous vs sequential turn structure in battle
  • Cosmetics-only vs convenience items in shop
  • Breed count limit per creature
  • Which AI art tool to standardize on
  • Whether death-marked creatures are tradeable on the marketplace
  • Soul Gem XP thresholds for each rank (F through S)
  • Whether Soul Gems are tradeable on the Steam marketplace
  • Class specialization names and tree structures for all three classes
