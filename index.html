Concept

A creature-collecting roguelite RPG where players climb an ever-ascending tower floor by floor. Each floor is a simple map where the Collector triggers battles until a win threshold opens the portal to the next floor, with manual chests scattered throughout. All metagame systems (breeding, crafting, farming, hatchery, crypt) are accessible as UI panels at any time during PvE — no physical base.

Game modes

Eternal
Soft death. Character stays in league, no reset. Houses the Eternal crypt (UI panel). Receives crypt contents from seasonal migrations.
Hardcore
On death, character is wiped to level 1 and stays in Hardcore. All items, creatures, and class XP (as soul gems) go to the Eternal crypt.
Seasonal
Soft death. Timed league. On season end, character migrates to Eternal. Seasonal crypt contents delivered to Eternal crypt.
HC Seasonal
On death, character wiped to level 1 and stays in HC Seasonal. Items go to Seasonal crypt. On season end, HC Seasonal crypt delivered to Hardcore crypt.

Ladders

  • PvP ladder — ranked 3v3 combat, tracked per season and all-time
  • Progress (floor) ladder — highest floor reached, tracked per season and all-time

Art pipeline

Dark-fantasy painterly style generated with gpt-image-2-2026-04-21 via OpenAI API (sizes 1024×1024 / 1536×1024 / 1024×1536 only). Generation model is gpt-image-2-2026-04-21 everywhere (creatures, floors, props, item icons) — it is sharper than every predecessor (~1.5–2× the edge sharpness of gpt-image-1.5, ~6× gpt-image-1); the legacy gpt-image-1/gpt-image-1.5 models are deprecated for generation. Transparency caveat: gpt-image-2-2026-04-21 does NOT support background:"transparent" (the API rejects it). Generate opaque on a flat neutral background whose value CONTRASTS the creature's dominant luminance, then obtain transparency via the remove.bg + despill pass below. Adaptive background rule (remove.bg segments by contrast — a same-tone background destroys the subject): dark/black-dominant creatures → flat WHITE or light-gray background. To force gpt-image-2 to actually render a light backdrop, describe the BACKGROUND as "a plain flat pure-white seamless backdrop" — but do NOT describe the LIGHTING as "studio / high-key / bright even lighting / product shot": that phrasing desaturates and flattens the creature (measured creature saturation drops to ~16–38 vs ~70 on a dark backdrop), producing washed-out photographic results that lose the dark-fantasy style. Background color and creature lighting are INDEPENDENT. Always keep the creature description on "richly saturated vivid colors, dramatic dark-fantasy rim lighting with deep shadows, moody painterly rendering, NOT washed out, NOT overexposed" regardless of background value; light/white-dominant creatures → flat dark-gray or black background; mid-tone creatures → mid-gray (#808080). NEVER strong chroma (green/magenta/pink) — they bake edge halos no pass can fully clean. After a white/light background, run a white-halo despill (darken over-bright edge-band pixels) in addition to the green/magenta despill. Native-transparent generation is only available on the deprecated 1.x models and must not be used. All creature art follows this pipeline:

Creature art pipeline (per line, 3 images)
  1. Stage 2 (core concept) — Fresh text-to-image generation via /images/generations on gpt-image-2-2026-04-21. Full descriptive prompt with stacked anatomical deviations (4–5 per creature), explicit color palette, combat-ready stance, generous framing margin, transparent or neutral-gray background. Prefer background: "transparent"; fall back to background: "opaque" with solid gray (#808080) or black (#000000) in the prompt if transparency causes edge issues. Never green or magenta/pink. Upload raw to codex.
  2. Stage 1 (infant) & Stage 3 (final form) — Stage 2 image sent as visual reference via /images/edits on gpt-image-2-2026-04-21. The prompt is a full descriptive brief for a new creature — never framed as "edit this image" or "make a smaller/bigger version" (that drifts/fuses anatomy). The image reference maintains visual lineage; the prompt drives a fresh generation. Same background approach as Stage 2 (transparent or gray/black — never green or magenta). Known limitation: the /images/edits endpoint ignores flat-background and center-70% framing instructions when handed a reference — it paints dark atmospheric scenery and fills the frame regardless of prompt emphasis. This is cosmetic only: the subject renders correctly and the matte pass strips the dark haze cleanly (~2–3% semi-alpha, no halo). If a truly clean solid background is required, use /images/generations (text-to-image, no reference), which respects background and framing — at the cost of the Stage-2 visual-lineage link.
  3. Background removal — Separate pass after art approval. Matte-only methods only — the method must never repaint the subject. Priority order: (1) remove.bg API — zero RGB drift, clean edges, commercially licensed only when processed with paid credits (subscription/PAYG); the 50 free calls/month return 500×500 previews that are not commercially licensed — quality-gate use only, never shipped. (2) BiRefNet general checkpoint (MIT) locally on the dev PC — zero drift, best on fine detail, free; use plain BiRefNet, never BRIA RMBG-2.0 (BiRefNet-based but non-commercial license — disqualified for a commercial Steam release). (3) rembg new_session("isnet-general-use"), plain, NO alpha matting — light fallback for simple silhouettes. Never gpt-image-1/gpt-image-2-2026-04-21 edits for removal (regenerates the whole frame — measured 12–17 RGB drift, warps anatomy) and never rembg default u2net (halos, destroys ethereal creatures). Despill pass (mandatory on legacy chroma art): creatures generated on the retired green/magenta backgrounds have colored fringe baked into their edge pixels — no matte can remove it since it is part of the subject. After matting, neutralize the cast in the partial-alpha edge band only (interior untouched): subtract magenta cast max(0, min(R,B)−G) or green cast max(0, G−max(R,B)), weighted by edge alpha. Fixes ~99% of legacy images in place without regeneration; only creatures whose soft ethereal edges (wisps, fine foliage) carried their color in the chroma bleed are regen candidates. New art on transparent/gray/black needs no despill. Originals preserved in codex/images/originals/ before overwrite (>1MB files: read via the git blobs API, not the Contents API inline field). BG_Status: has_chroma → bg_pending → bg_approved or manual.
  4. Crop option slots — the codex shows each creature's candidate cutouts as sequential tabs: A = codex/images/<ID>.png (main/original), B = codex/images/cropped/<ID>.png, C = codex/images/cropped_c/<ID>.png, D = cropped_d/, etc. A new option ALWAYS fills the next empty slot in order (A→B→C→D). Never skip a slot — the codex will not render a C when B is empty. If only A exists, a new crop goes to B; if A+B exist, it goes to C. On approval, the chosen crop is promoted to A (main) and all other slot files are deleted so no dead tabs remain. Slots are contiguous and RESHUFFLE on deletion: deleting any slot shifts every higher slot down to fill the gap — delete A → B becomes A, C becomes B, D becomes C; delete B → C becomes B, D becomes C. There is never a gap (no C without B, no A missing while B exists). Physically this means moving each higher slot's file content down one path (cropped/→main, cropped_c/→cropped/, …) and deleting the now-empty trailing path.
  5. CSV update — Image_Path and Art_Status set immediately after each generation. Statuses: pending (awaiting approval), approved (art locked, do not regenerate), redo (flagged for regeneration).
Floor art pipeline (per floor, 3+ assets)
  1. Battle background — Text-to-image, 1536×1024 (regular floors) or 1024×1024 (boss arenas). Prompt includes zone visual theme, encounter type, and atmosphere. Transparent background not used — these are full scene paintings. Tracked in codex/floor_assets.csv.
  2. Parallax layer — One per zone (10 total), 2304×1024, seamless horizontal tile. Atmospheric depth layer behind the battle background for ultrawide displays.
  3. Props — 3–4 per zone, transparent background. Environmental dressing placed on the floor map: standing stones, skull heaps, glowing crystals, etc. Tracked in codex/prop_assets.csv.
  4. Map preview — Two renders per floor: empty (background + props, no creatures) and populated (with creatures placed). For approval only, not shipped assets. Tracked in floor_assets.csv Map_Preview_Status / Map_Populated_Status.

Floor assets live in floors/backgrounds/, floors/parallax/, floors/props/{zone}/, and floors/previews/. The floor sheet (codex/floor_assets.csv) references which assets each floor uses — decoupling floor identity from art allows infinite floor extension later by reusing zone asset pools.

Item art pipeline (per base type, 1 image)
  1. Generation — Text-to-image, 1024×1024, transparent background. Each base type (weapon, armour, accessory, tag, material) gets one icon. Enhancement tiers (Rough → Sublime, Faded → Elder, Tattered → Fated) each get a tier-specific prompt describing the item's visual state at that power level.
  2. Approval — Same status flow as creatures: prompt_ready → generated → approved. Tracked in codex/items.csv Art_Status column.

All 435 items catalogued in codex/items.csv with flavor text and image prompts. Prompt boilerplate: isolated dark-fantasy game item icon, oil-painted style, dramatic rim lighting, rich vivid colors, item centered 70% of frame, three-quarter angle, transparent background.

Character system

One character per league, always

Every account has a character in every league they have unlocked, starting at level 1. The character slot is permanent — death does not remove or demote the character. On hardcore death, the character is wiped back to level 1 and remains in the same league. Only their items, creatures, and class XP are lost.

Character level vs class level

The character has a character level — a global level aggregated from XP across all classes. Character level scales base stats (HP, stamina, resistances). Each class has its own class level tracked separately, unlocking skill tree nodes and class-specific bonuses. Switching classes does not reset others.

Skill tree

Each class has a Path of Exile–style passive skill tree — a large interconnected graph of nodes. Players spend skill points (earned on class level-up) to unlock adjacent nodes only.

  • Node types: stat (flat bonus), modifier (% bonus), keystone (build-defining passive)
  • Must path from starting node outward — no jumping to isolated nodes
  • Respec costs resources but is always possible
  • Class specializations gate deeper sub-trees inaccessible without specialising

Soul gems

When a hardcore character is demoted, their soul fractures. Each class they trained echoes forward as a Soul Gem, deposited in the crypt for the surviving character to reclaim.

One Soul Gem is generated per class the character had any XP in. Rank scales with XP amount. Consuming a gem grants that XP to the character's matching class.

The curve is deliberately back-loaded for an autoplay game: 75% of all leveling time sits in levels 90–100 (50% in the last five levels). Total to max level: ~20,300 battles ≈ 680 h at 2 minutes per battle. Reaching level 67 — where Stage 3 creatures and final evolutions unlock — takes only ~885 battles (~30 h).

Hardcore implication: death wipes to level 1 — the relevel back to 67 is ~30 h of autoplay, but a death at level 95+ costs a dedicated player weeks. High stakes by design.

There is no XP multiplier bonus for fighting creatures above the player’s level. The only incentive to push harder content is the higher flat XP that higher-level creatures naturally grant (a level 40 creature gives 400 XP vs 300 from a level 30 creature). This keeps the system clean and unexploitable.

Zone creature level distribution

Creatures on each floor spawn within ±3 levels of the zone’s base level, following a symmetric bell distribution. The weighted average XP per kill equals the zone-level XP exactly, preserving the timing math above.

Offset from zone level−3−2−10+1+2+3
Spawn rate3%7%12%56%12%7%3%

Average preservation: Σ(rate × offset) = 0.03(−3) + 0.07(−2) + 0.12(−1) + 0.56(0) + 0.12(+1) + 0.07(+2) + 0.03(+3) = 0. Average XP per kill = 10 × zone_level.

Core loop

Single active character

The game runs one league character at a time. Switching leagues switches the active character; only the active character simulates. There is no offline progression; being in-game is the only way to advance. All economy balancing is denominated in battles for this one active character (~30 battles/hour on autoplay).

Manual mode
Default state. Player directly controls the active character. Autoplay must be explicitly enabled to hand off control.
Autoplay mode
Opt-in. The active character grinds the current floor automatically. Disable to return to manual control.
  • Autoplay is off by default — the player opts in
  • Switching leagues pauses the previous character entirely
  • Closing the game window pauses the character
  • No offline rewards — open game only

The earlier multi-tab simultaneous-simulation design (up to 4 league tabs at once) is retired — one character farms at a time.

Floor presentation (Simplified Static)

There is no player avatar. Each floor is a static painted map (1536×1024 px, 3:2; parallax background at 1.5× for ultrawide; boss floors are 1024×1024 arenas). Creatures wander the map autonomously; hovering a creature highlights it by rarity; clicking (tagging) it starts the battle. Grid movement is archived — see Roadmap.

Map design

Maps are small and simple — no elaborate handcrafted circuits or secret areas. Battles start the same way on every floor: click a wandering creature. Floor themes are purely visual — they change the art, not the mechanics.

🌿 Outdoors
Visual theme only — open-air backdrop, grass and foliage dressing.
🕳️ Cave
Visual theme only — enclosed rock backdrop, darker palette.
🎣 Fishing
Visual theme only — waterside backdrop, aquatic dressing.

Floor theme is fixed per floor and purely cosmetic. The battle trigger (click a creature), win condition (consecutive win streak scaled by floor depth), boss key drop, and chest system are identical across all three themes.

🏆 Win condition
Win a streak of consecutive battles on a floor to open the portal. Threshold scales with floor depth: ceil(floor0.65) — floor 1 needs 1 win, floor 10 needs ~4, floor 50 needs ~13, floor 100 needs ~20. Losing a battle resets the streak to zero. Boss floors always require exactly 1 battle (the boss fight).
👑 Boss floors
Every 10th floor is a boss room (floors 10, 20, … 100). Entry requires consuming a Key, which has a chance to drop from any battle across the previous 9 floors. Guaranteed key: the first time a player completes floor 9 of any zone (floors 9, 19, 29, … 99), they are guaranteed a key drop for that zone's boss. This is a one-time reward per zone per character — subsequent visits use the normal random drop rate.

Streak counter

After every battle, a compact rectangle appears mid-screen showing the current streak status as X / Y (battles won / threshold). On a win, X scrolls up to the new count and the text pops in size briefly. On a loss, the rectangle visually cracks and X rolls back down through each number to 0. The counter only appears on floors not yet cleared.

Boss key

A single Key is required to enter a boss floor, accessible via the portal or directly through the map selector. Any battle won on the 9 floors preceding a boss has a chance to drop that zone's Key. Guaranteed key: the first time a player completes the 9th floor of a zone (floors 9, 19, 29, … 99), a key is guaranteed to drop. This is a one-time reward per zone per character. The Key is consumed on boss entry.

Chests

Chests are scattered around each map as manual-only, one-time pickups — autoplay does not collect them. Each chest can only be opened once per floor visit cycle. The map selector displays chest collection progress per floor as x/x collected.

All equipment and crafting materials now come exclusively from battle rewards and chests — there is no resource gathering layer. Gathering nodes have been removed from active scope; see Roadmap for details.

Defeat

Losing a battle does not remove the player from the floor. The team remains on the current floor and the win streak resets to zero. There is no forced exit, no floor lock, and no return to floor select.

Autoplay

Autoplay is disabled by default and must be enabled per character. A quick toggle sits directly on the gameplay screen; the Auto tab in the options menu holds the full customization. It can be enabled on any tab including the currently selected one. To take manual control of a character, disable autoplay on that tab.

Loop: every few seconds after exiting a battle, autoplay tags a random creature on the map while the camera gently pans over to it, resolves the battle, then repeats — advancing or descending floors according to the options below.

Auto advance floors
When the win streak threshold is met and the portal opens, automatically advance to the next floor. Off — keep farming the current floor.
On death
( ) Stay — remain on the current floor and rebuild the streak. ( ) Go down — drop one floor and continue there.

Healing

Healing items have been removed from the game entirely — there are no consumable heals, in or out of battle. The team is restored to full HP after every battle, win or lose. Attrition therefore exists only within a single battle; difficulty pressure comes from the streak reset on defeat, not from resource depletion between fights.

Turn structure

Simultaneous selection, sequential resolution. Both sides select actions blind (double-blind). Once locked, all actions resolve in Speed order — highest Speed acts first. Status effects queued (poison, stun, burn, etc.) resolve after move resolution. This makes Speed the decisive tactical axis: a faster creature can KO a threat before it acts, even if both sides chose to attack the same target.

PvP sync

Both players submit actions simultaneously. Server resolves outcomes. Unity Relay for connection. Never resolved client-side.

Anti-cheat: Server-authoritative or server-verified only for PvP.

Combat stat axes

4 axes per creature and character. No Physical/Special split — single damage lane. Grade (1–10) determines base stat value identically across all species: a 6/10 Offense creature always has the same base Attack as any other 6/10 creature at the same level.

AxisRoleScaling
OffenseDetermines damage dealt by attacksScales with level
DefenseReduces incoming damage; determines HP poolScales with level
UtilityStatus effect potency & duration, healing/shield powerScales with level
SpeedDetermines turn order — highest Speed acts firstFlat — does not scale with level

Base stats are determined entirely by axis grade + level, multiplied by creature quality grade. Species identity comes from type, moves, and grade distribution (which axes are high vs low), not from stat values. Full numerical tables for all equipment and modifier systems live in the Stat & Equipment Tables page.

Damage formula

RawDamage = ((2 × Level / 5 + 2) × MovePower × Offense / Defense) / 50 + 2

FinalDamage = RawDamage
            × TypeMod × STAB × CritMod × Random(0.85–1.0)
            × (1 + Σ increased%)
            × Π(more multipliers)

The base formula follows the proven Pokémon/Coromon core. The increased/more split (from Path of Exile) is bolted onto the modifier chain — increased sources add together (diminishing returns within a category), more sources multiply (always valuable). This rewards diversifying modifier sources over stacking one type.

Creatures have a fixed set of moves tied to their species and theme. Moves are not swappable or customizable.

Metagame systems

All metagame systems are accessible as UI panels at any time during PvE floors, including during battles. They are locked during PvP. There is no physical base — everything is menu-driven.

🧬 Breeding
Two parent creatures fed Gruel produce an egg via UI. Higher grade creatures require more Gruel. Higher grade parents increase odds of higher grade eggs. No grade cap on bred eggs.
🔧 Crafting
Recipe list in UI. Select recipe, confirm materials, receive output. Materials sourced from battle rewards and chests.
🌱 Farming
Timer-based UI system. Plant a crop, return after a set duration to harvest. Yield renders into Gruel or crafting materials.
🥚 Hatchery
Egg incubation managed via UI. Set eggs to incubate, return after the timer. Hatchery level (upgraded via crafting resources) improves grade odds and slot count.

Creature acquisition

There is no capture mechanic. All creatures come from eggs — either dropped in combat or produced via breeding. Every egg must be incubated in the Hatchery panel before hatching. Egg grade determines the stat range of the hatched creature and becomes the creature's grade.

🥚 Wild eggs
Drop from combat encounters. Grade range: Worn through Legendary only — no wild eggs above grade 6. Grade determines stat range at hatch.
🧬 Bred eggs
Produced via the Breeding UI panel by selecting two parent creatures and spending Gruel. No grade cap — can produce up to Void. Higher grade parents significantly increase odds of higher grade output. Hatchery level also increases the chance of higher grade eggs independently of parent grade.

Creature souls

Each piece of character gear can have the soul of a creature attached to it, granting special effects. Every creature has a different effect for each gear slot. Souls have one of three scopes:

Soul scopeEffectExample
LocalBuffs only that item's contribution — strongest per-item impactThornback on Main Hand: "This weapon's Offense ×1.20"
PersonalBuffs the character's total stats — moderate power, affects all actionsDusthound on Gloves: "Character +8% crit chance"
TeamBuffs all allies (character + both creatures) — weaker per target but ×3 entitiesYggdrasil on Boots: "All allies +6% Defense"

Soul scope varies per creature per slot. The same creature's soul can be Local on one slot and Team on another, creating targeted build decisions around which soul goes where.

Modifier scopes

Every modifier in the game has one of three scopes, determining what it affects.

ScopeAffectsUsage
LocalOnly the item it's on — scales that item's own stat contributionDefault for all character gear modifiers (gems, runes, enchantments, refinement). A rune's +8% on a weapon only multiplies that weapon's base + gems.
PersonalThe bearer as a whole — the character or the specific creature wearing itSkill tree nodes. Creature tag modifiers (tags are the exception — all tag modifiers are Personal, affecting the creature's total stats). Some creature souls.
TeamAll allies (character + both creatures)Some creature souls. Some skill tree keystones. Individually weaker per target but multiplied across 3 entities.

Modifier slots

💎 Gems — flat stats
Fixed flat modifiers. 6 gem types: Garnet (+Offense), Onyx (+Defense), Bloodstone (+HP), Sapphire (+Status Potency), Ruby (+Crit Damage), Topaz (+Move Power). Set at socket time, removal destroys the gem. On character gear: Local scope. On Tags: Personal scope. Up to 3 slots at Void grade.
🔷 Runes — percentage scaling
Fixed percentage modifiers. 5 rune types: Ruin (+% Offense), Aegis (+% Defense), Vitality (+% HP), Precision (+% Crit Chance), Affliction (+% Status Duration). Multiply the item's base + gems (Local) or the creature's total (Personal on Tags). Set at socket time, removal destroys the rune. Up to 3 slots at Void grade.
📜 Scrolls — random modifier pool
Socketable items for enchantment slots. On socket, the scroll rolls a random modifier from the pool — flat, percentage, or unique effects (crit chance, crit damage, status resist). Rerollable for Ink. Up to 3 slots at Void grade.

Enhancement items — tiers, grades & materials

Gems, runes, and scrolls are items in their own right: each has a level tier (10 named tiers matching the 10 level brackets — e.g. Rough → Cut → … → Radiant → Sublime for gems) and a grade (Worn–Void). An enhancement's tier must be ≤ the gear's tier to socket. Full tier ladders and all values live in the Stat & Equipment Tables.

FamilyDismantles intoMaterial pays for
💎 GemsDustCrafting new gems, grading gems up
🔷 RunesEssenceCrafting new runes, grading runes up
📜 ScrollsInkCrafting new scrolls, grading scrolls up, rerolling a socketed scroll's modifier (3 × tier Ink)
  • Dismantle yield: tier × grade# material (a T10 Rare gem → 40 Dust)
  • Craft: 6 × tier material produces a Worn enhancement of chosen type at that tier — the junk-to-on-tier conversion loop
  • Grading up uses the same success ladder as gear (paid in the family material instead of Ichor)
  • Removing a socketed gem or rune destroys it; the socket is freed
  • Materials (Dust / Essence / Ink) are freely tradeable commodities; enhancement items follow the grade-5+ trade gate

Refinement

Nothing is useless

Every drop has value, however marginal. All gear dismantles into Scrap (plus Ichor at Rare+), and Scrap crafts the next tier of equipment. All enhancements dismantle into their family material (Dust / Essence / Ink), which crafts and grades the next tier of enhancements. A Worn Rough gem drop at floor 95 is still Dust in the pipeline.

Gear crafting & level requirements

RuleDetail
Craft gear25 × tier Scrap produces a Worn item of chosen slot and base type at that tier (T10 = 250 Scrap)
Equip requirement — character gearTier N requires character level ≥ (N−1)×10 (T1 = level 1, T2 = 10, … T10 = 90). Gear above the character's level cannot be equipped.
Equip requirement — TagsSame thresholds against the creature's level. A creature cannot wear a Tag of a higher tier than its own level bracket.
Enhancement socketingEnhancement tier ≤ the gear's tier (unchanged)

Any piece of gear can be refined from +1 up to +10, increasing its power. Refinement uses Scrap, acquired by dismantling equipment. Refined equipment yields more Scrap when dismantled, creating a recycling loop.

RuleDetail
CurrencyScrap — gained by dismantling gear
Cost scalingHigher equipment tier and higher current refinement both increase Scrap cost
Success chance+1→+10: 100 / 90 / 80 / 65 / 50 / 35 / 20 / 10 / 3 / 1% — no pity, pure roll every attempt
Attempt cost3 × target level × item tier Scrap (T10 +10 attempt = 300 Scrap)
On failureNo progress, no destruction — Scrap is consumed but gear is unaffected
Dismantling refined gearYields more Scrap than base gear — scales with refinement level

Grading up

Gear can be upgraded to the next grade (e.g. Rare → Epic) using Ichor. Functions identically to refinement in terms of cost scaling, failure chance, and no-destruction on failure.

RuleDetail
CurrencyIchor
Cost scalingHigher current grade increases Ichor cost
Success chanceInto C/UC/R/E/L/A/D/P/V: 100 / 90 / 70 / 50 / 35 / 20 / 8 / 3 / 0.5% — no pity, pure roll every attempt
Attempt cost1 / 1 / 2 / 3 / 5 / 8 / 12 / 18 / 40 Ichor per attempt by target grade
On failureNo progress, no destruction — Ichor is consumed but grade is unchanged

Gear combining

3 × same-slot, same-grade gear combine into 1 random base of that slot at the next grade — capped at Epic. Grade 5 (market entry tier) is deterministically grindable; Legendary and above only come from the Ichor gamble and lucky drops, protecting high-end market value.

Economy & marketplace

Acquisition & upgrade economy

All progression currencies flow from battles and chests, denominated in battles (~30/hour on autoplay, single active character). Journey milestones are near-deterministic (thousands of drop rolls); the endgame is gated by pure-RNG success rolls with no pity system anywhere — the market is the variance smoother. Full numeric tables in Stat & Equipment Tables.

IncomeRate
Gear drop10% per battle, at-floor tier, grade weights by floor band (Divine+ never drops)
Enhancement drop (gem / rune / scroll)10% per battle, even family split, grades one band behind gear
Scrap~150/h endgame (dismantling)
Ichor~10/h endgame (bosses, Rare+ dismantles, deep-floor drops, chests)
Dust / Essence / Ink~20/h per family endgame (enhancement dismantles, chests)

The endgame chase — expected / lucky / unlucky (no pity)

Success gates are geometric distributions: the spread between a lucky (p10) and unlucky (p90) player on a single gate is ~22×. Failure costs currency only — no destruction, no downgrade. Raw chances: Void 0.5%, +10 refinement 1%.

One item, Worn→Void gradingOne item, 0→+10 refinement
Expected ~8,800 Ichor ≈ 880 h · p10 ~2,000 · p90 ~19,000 (Void gate alone: 841 / 8,000 / 18,400)Expected ~43,700 Scrap ≈ 290 h · +10 gate alone: p10 10, median 69, p90 229 attempts at 300 Scrap each

A marquee item taken to Void +10 is ~1,200 h expected — a year-scale chase at 3–4 h/day, months even on 24/7 autoplay, with honest odds of doubling that. A full 12-slot Void +10 loadout is a multi-year, market-aggregated pursuit; Void enhancements (expected ~282,000 material each) are mythical trade goods. Casuals buy from the market; grinders and lucky rollers supply it — that variance is what gives grade-5+ items real price discovery.

Monetization model

  • Free to play on Steam
  • Microtransactions via Steam / Unity Economy
  • Battle pass per season
  • Steam takes 5% market fee + configurable game fee (up to ~10%)

Cosmetics-only monetization is safer for player reception and avoids regulatory issues in certain markets. Decide scope early.

Critical risks

PvP anti-cheat — server-authoritative resolution required. Design this before any PvP work begins.

Art consistency — lock style-guide prompt before mass generation.

Death pipeline integrity — four death paths + two crypt systems + season migration must all be tested exhaustively. HC Seasonal double-migration is highest risk.

Seasonal migration — crypt delivery mechanism must be rock solid before launch.

Trade duplication exploits — implement transfer lock before marketplace goes live.

Future features (cut from current scope)

Rogue Collectors — hostile NPC trainer encounters with patrol routes and line-of-sight triggers, originally part of floor structure. Removed from active scope to keep early development focused. May be reintroduced post-launch or in a content update.

Gathering nodes & secret areas — the original handcrafted circuit map design with resource nodes, HM-gated secret areas, and a node-farming layer has been replaced by the simpler battle-win-threshold map design. All equipment and crafting materials now come from battle rewards and chests only. May be reconsidered as a future map variant or content layer.

Physical base — the original design included a physical base the Collector could walk around in, with tile-based farming, craftings stations, a hatchery building, and a crypt chest to interact with. This has been replaced entirely by UI panels accessible during PvE floors. A physical base may be reintroduced in a future content update as an optional social/cosmetic space.

Isometric Exploration — the original prototype used top-down Gen II–style grid movement with a player avatar, tile-based encounter triggers (grass/cave/fishing), and autoplay pathfinding. Archived in favor of the Simplified Static approach for faster prototyping. The full grid-movement system can be restored from the archived roadmap above.

Zone bosses

Every 10th floor (10, 20, … 100) is a zone boss and has no wild encounters. A boss is a full team like the player's — a character plus 2 active creatures and 4 in reserve — not a wild creature. The SSR flagship creatures (Dagon, Ouroboros, Kraken, …) are rare wild encounters on ordinary floors, not the bosses.

Bosses scale up considerably each zone: floor 10 is the weakest, floor 100 the hardest of the base tower. Definitions live in codex/bosses.csv (team fields currently TBD). Post-level-100 content — deeper floors with fully-optimized boss teams — is out of scope for now and noted for later.

Creature grading design principle

Each creature line has one fixed set of 4 identity grades — Offense / Defense / Utility / Speed, scale 1–10 — that stays constant across every evolution stage in that line. Evolution scales actual battle stats later; it does not change the identity shape.

Creature naming

Every creature line name is a single word — no multi-word names, no "The ___". Dark-fantasy tone, frequently a modified myth or religious reference. Line_ID in the CSV is the uppercase line name. The three evolution stages now carry distinct names: Stage 3 keeps the line name, while Stages 1 and 2 are real-word precursor names drawn from the final name's own myth, etymology, or life-cycle (e.g. Veliger → Hafgufa → Kraken; Eyas → Haggard → Roc). Applied across all 100 lines in codex/creatures.csv. Line_ID stays the final name, so evolution links and encounter IDs are unaffected.

Image generation conventions

  • Background & removal rules: see the canonical Art pipeline section (Overview) — the single source of truth. In short: generate on transparent or gray/black, never green or magenta (baked-in edge fringe); remove matte-only (remove.bg paid / BiRefNet local), then run the despill pass on any legacy chroma art.
  • 3-stage workflow: see canonical Art pipeline section. In short: Stage 2 first (text-to-image), Stages 1 & 3 as fresh generations using Stage 2 as visual reference — never an "edit" instruction.
  • Every prompt stacks 4–5 anatomical deviations so the creature reads as its own species, not "real animal + one gimmick."
  • Every prompt names an explicit color palette. No two lines in the same zone share a palette.
  • Creatures must be in a natural combat-ready stance (this is a turn-based RPG battle system).
  • Full body must fit within the frame with generous margin — nothing cropped at edges.
  • Reframe fix for edge-cropped creatures: If a creature's anatomy is cut off by the frame edge, regenerate with aggressive center-framing: The creature occupies only the CENTER 70% of the frame, surrounded by clear empty background on all sides — nothing even close to any edge. Do NOT use outpainting/edits to extend — the edits endpoint degrades detail quality across the whole image even when only borders are masked. Always regenerate fresh.
  • Prompt boilerplate: In a natural combat-ready stance as if facing an opponent. Full body visible with generous margin on all sides, nothing cropped at edges. Centered, painterly dark-fantasy illustration, dramatic rim lighting with clear mid-key exposure, vivid readable colors, one single cohesive creature, no text, no props, no scenery. No floating or detached parts.
  • No repeated core motifs between different creature lines.
  • Stage-2 prompts stored in Image_Prompt column of codex/creatures.csv.
  • Approved art is locked — never regenerate or rewrite prompts for approved creatures without explicit instruction.
